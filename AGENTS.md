# AGENTS.md

本文件定义本仓库的默认协作约定，作用域为仓库根目录及其所有子目录。更深层目录如存在新的 `AGENTS.md`，则以更深层文件为准。

## 1. 目标与原则

- 以可维护性、可恢复性、可观察性为优先，不为了短期方便牺牲长期结构。
- 优先做小而清晰的改动，避免一次性引入过多抽象。
- 新增能力时，先复用现有结构与约定，再考虑新增概念。
- 任何运行时行为都应尽量可解释、可追踪、可落盘。

## 2. 技术栈约定

- 当前项目默认使用 TypeScript，并采用 Node.js ESM 风格。
- 优先使用社区成熟、维护良好的开源工具，而不是从头实现通用基础设施。
- 只有当现有开源方案明显不适配当前需求，才考虑自研替代。
- 新增第三方依赖前，应先确认其必要性、维护状态、社区活跃度以及与当前架构的兼容性。

## 3. 目录与文件约定

- `src/` 存放源码。
- `.codex/` 存放 Codex 原生能力相关内容。
- `.loong/` 仅用于运行时数据，不应承载业务源码。
- `agents/` 用于表达组织树结构；如后续由 `loong init` 生成，需保持递归一致性。
- 文件夹和文件名应简洁为主，统一使用小写单词命名，如果有多个单词，则单词间使用 `-` 连接。
- 避免使用驼峰式文件名、PascalCase 文件名或下划线命名作为默认规则。
- 每个源码目录下都应创建 `index.ts`，用于统一导出该目录下的其他模块。
- 跨目录引用时，优先通过目录级 `index.ts` 暴露的稳定导出访问内部模块，而不是随意深层直连。

## 4. 编码风格

- 默认优先采用面向对象的编码方式组织核心逻辑，特别是具备明确职责边界的模块。
- 当某段逻辑天然体现为“实体 + 职责 + 生命周期”时，优先使用类来承载，而不是堆叠过程式函数。
- 工具型纯函数仍可保留为函数，但不应让核心流程退化为无边界的脚本式代码。
- 类的方法名默认统一使用小写字母加下划线命名；文件名和目录名仍保持小写单词加 `-` 的规则。
- 默认使用清晰、直接、可读的实现，不炫技。
- 变量、函数、文件名应优先表达业务含义，而不是实现细节。
- 单个函数应尽量只做一层明确的事；过长函数应主动拆分。
- 尽量避免隐式共享状态；状态读写应明确集中。
- 对关键边界、状态转换、恢复逻辑，允许添加少量解释性注释。

## 5. 运行时与状态

- 状态、日志、记忆、运行绑定应有清晰边界，避免互相污染。
- 不允许父节点直接代管子节点完整内部状态；只保留必要监督事实。
- 新增持久化数据时，应先明确其语义：
  - 是权威状态
  - 还是派生数据
  - 是否可重建

## 6. CLI 设计约定

- CLI 命令应保持直观、稳定、少惊讶。
- 错误提示应可读，避免只输出技术性异常堆栈。
- 非必要不在命令执行中引入交互式阻塞。
- 能后台运行的能力，优先按守护式和可恢复方式设计。

## 7. 测试与验证

- 每次改动至少要做与改动范围匹配的验证。
- 本项目是围绕 Codex 的薄运行时层，测试应优先守住运行协议与文件契约，而不是追求复杂的传统分层或覆盖率形式。
- 测试体系应保持简单，默认采用：
  - `tests/runtime/` 存放主力运行时契约测试，可使用临时目录、真实文件系统和 fake Codex 响应
  - `tests/smoke/` 存放真实 CLI 入口冒烟测试，以及显式开启的 Codex canary
  - `tests/helpers/` 存放共享测试辅助工具
- 非必要不拆分 `unit`、`integration` 等复杂测试层级；只有当执行策略、速度或排障确实需要时再增加分类。
- 测试应验证稳定契约和用户可观察行为，而不是绑定当前实现细节。避免为了已有代码形状倒推测试用例，尤其不要断言无业务含义的内部调用顺序、mock 调用次数、临时文案、私有辅助函数拆分、可选参数是否为 `undefined` 等脆弱细节。
- 文案类变更通常不应新增或修改测试，除非该文案本身就是明确的对外协议、错误码、CLI 输出契约或提示词契约。能用更高层行为表达的测试，不要锁死具体措辞。
- 当测试必须使用 fake、mock 或 spy 时，应只用它们制造边界条件或观察必要副作用；断言重点应落在状态落盘、文件变化、恢复路径、返回结果和外部可见事件上。
- 对提示词驱动的能力，优先补充 prompt contract tests，验证提示词中是否稳定包含关键运行规则、文件格式、返回 JSON 契约和当前状态注入。
- 真实 Codex canary 应保持可选，必须通过环境变量显式开启，不进入默认必跑链路。
- 优先验证：
  - 命令是否能运行
  - 目录骨架是否正确生成
  - 状态是否正确落盘
  - 恢复逻辑是否符合预期
- 如暂未补测试，应在说明中明确指出验证缺口。

## 8. 文档约定

- 新增重要概念时，同步更新文档或计划文件，避免知识只留在代码里。
- 文档优先说明“为什么这样设计”，其次才是“怎么实现”。
- 如果实现偏离已有计划或规格，应同步修正文档，而不是让代码和文档长期分叉。

## 9. Git 与变更约定

- 不做无关重构。
- 不顺手修改与当前任务无关的文件。
- 保持提交范围清晰：结构调整、功能新增、文档更新尽量可区分。
- 运行时生成物不应混入源码提交，必要时更新 `.gitignore`。

## 10. 决策优先级

当存在多个可行方案时，优先顺序为：

1. 是否符合当前规格与计划
2. 是否复用现有结构
3. 是否更容易维护与恢复
4. 是否更容易观察与排障
5. 是否实现更简单

## 11. 默认协作行为

- 对未明确的实现细节，优先采用保守、可解释、易回滚的方案。
- 对可能扩大范围的改动，先固定边界，再实现。
- 对尚未收敛的架构问题，先补文档和计划，不急于编码。

## 12. LLM 编码行为准则

以下准则用于减少常见的 LLM 编码错误。

### 12.1 编码前先思考

不要假设。不要隐藏困惑。要暴露取舍。

实现前：

- 明确陈述自己的假设。如不确定，应询问。
- 如果存在多种理解，应列出来，而不是悄悄选择一种。
- 如果存在更简单的方案，应说明。必要时提出反对意见。
- 如果某件事不清楚，应停下来，指出困惑所在，并提问。

### 12.2 简单优先

用能解决问题的最少代码。不要做推测性的扩展。

- 不添加超出请求范围的功能。
- 不为单次使用的代码添加抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果写了 200 行而本可以用 50 行解决，应重写。

应自问：“资深工程师会不会认为这过度复杂？”如果答案是会，就简化。

### 12.3 外科手术式改动

只触碰必须触碰的内容。只清理自己造成的问题。

编辑既有代码时：

- 不“顺手改进”相邻代码、注释或格式。
- 不重构没有坏掉的东西。
- 匹配既有风格，即使自己会采用不同写法。
- 如果发现无关的死代码，应提及它，但不要删除。

当自己的改动造成孤立内容时：

- 移除由本次改动造成的未使用 import、变量或函数。
- 除非被要求，不移除改动前已经存在的死代码。

检验标准：每一行变更都应能直接追溯到用户请求。

### 12.4 目标驱动执行

定义成功标准。循环推进直到完成验证。

将任务转化为可验证的目标：

- “添加校验” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写能复现该 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，应陈述简短计划：

```text
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强成功标准可以让执行独立循环推进。弱成功标准（例如“让它能工作”）需要持续澄清。

这些准则如果生效，应体现为：差异中更少出现不必要变更，更少因过度复杂而重写，并且澄清问题发生在实现之前，而不是错误之后。

## 13. 近期协作偏好补充

- 提示词设计应尽量采用代理自身视角描述规则与环境，不向代理暴露“框架会自动处理”之类的内部实现细节；优先描述它能看到什么、应如何行动。
- 在满足需求的前提下，优先选择更直接、更简洁的实现，避免为了少量状态管理引入额外的全局状态文件、包装层或返回结构。
- 如果某个状态天然隶属于某个具体工作单或目录，优先把标识落在该工作单目录内，而不是集中维护一份额外的全局状态。
- 函数接口应尽量收敛，避免为了传递临时上下文扩大返回类型或增加冗余参数；能在函数内部直接获取的信息，就不要额外透传。
- 类似“本轮前收集状态、本轮后更新状态”的流程控制，优先放在运行循环或 orchestration 层处理，而不是把流程性状态耦合进纯提示词构造函数。
- 对工作单不可修改这类约束，如用户已明确接受提示词约束，则默认不额外升级为运行时强约束，除非后续明确提出需要更强保障。
- 代码组织粒度应取中：避免把模块全部堆回少数大文件，也避免按 `repository/service/builder/tracker` 之类模式过度细拆。
- 优先让目录表达职责边界，再由目录内少量文件承载实现；单个职责目录下的文件数量应尽量克制，避免继续演化成新的杂物间。
- 命名应短、直接、稳定，优先使用领域词汇，不为“架构好看”引入过长或过泛的名字。
- 如果某个领域已经存在明确的管理类或核心类，对外能力应优先收口到该类，由上层通过统一入口访问；同领域能力不要同时长期分散在类方法和一组平级导出函数之间。
- 测试设计偏好务实、轻量：不为了“看起来标准”而引入复杂目录或概念；如果项目逻辑主要来自提示词注入和文件契约，应把测试重点放在运行协议、状态落盘、工作单同步、提示词契约和 CLI 冒烟上。
- 当提出方案后如已形成明确目录或协作约定，后续实现应严格遵守；如发现约定不适合当前项目，应先说明原因并主动收敛为更简单的结构。
